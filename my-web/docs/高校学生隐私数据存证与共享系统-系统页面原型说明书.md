### 1.3 设计定位

## 2. 设计目标与原则

### 2.1 页面设计目标
- 支持四级权限角色清晰分流：学生、辅导员、教务管理员、系统管理员。
- 围绕数据存证、授权共享、追溯审计、隐私保护四大核心模块建立连续业务路径。
- 页面结构兼顾论文展示性与真实后台系统可用性。
- 通过统一视觉语言和反馈机制提升用户友好度。

### 2.2 界面设计原则
1. **可信优先**：链上摘要、状态标签、审批过程、访问留痕必须清晰可见。
2. **最小暴露**：敏感字段默认脱敏，必要时二次确认后查看。
3. **高频优先**：常用操作在首屏可达，如新增数据、发起存证、处理审批。
4. **低负担交互**：表单分步、筛选就近、错误提示就地。
5. **角色自适应**：同一系统下不同角色看到不同首页卡片与菜单重点。
6. **可演示性**：适合在答辩时快速讲清“提交—审批—上链—授权—追溯”的闭环。

## 3. 全局视觉与交互规范

### 3.1 视觉风格建议
- 主色：**深蓝色**，用于强调可信、规范与平台感。
- 辅助色：**青绿色**，用于表示安全、通过、有效。
- 状态色：待处理使用橙色，异常/拒绝使用红色，普通说明使用灰蓝色。
- 风格关键词：**学院治理感、数字可信、清爽留白、卡片化信息呈现**。

### 3.2 布局建议
- PC端优先，推荐 1440 像素设计稿宽度。
- 统一采用 `顶部栏 + 左侧导航 + 主内容区` 结构。
- 主内容区优先使用“筛选栏 → 摘要卡 → 表格/内容区 → 抽屉/详情”的节奏。
- 详情型页面推荐“左主右辅”布局，左侧放业务内容，右侧放摘要、状态、时间线。

### 3.3 组件建议
- 统计卡片
- 状态标签
- 步骤条与时间线
- 抽屉式详情
- 分步表单
- 上传面板与进度条
- 哈希/交易信息复制组件

### 3.4 用户友好性要求
- 错误提示尽量靠近输入项，不使用过度打断的弹窗。
- 空状态页面给出下一步操作引导。
- 长哈希与长编号支持复制按钮。
- 高风险操作（撤销、作废、下载高敏文件）需要确认说明。
- 页面上尽量使用业务语言，而不是把链上技术字段直接扔给普通用户。

## 8. 页面设计落地建议

1. 建议采用 Vue 3 + Element Plus / Ant Design Vue 进行后台风格页面实现。
2. 所有列表页优先使用统一表格组件，保证筛选栏、分页、状态标签风格一致。
3. 高敏信息默认脱敏，查看原文需要单独确认。
4. 统一使用时间线组件表达“申请—审批—上链—授权—访问—追溯”链路。
5. 对区块链技术信息进行二次业务翻译，如“区块高度”“交易哈希”“上链时间”，避免页面只适合技术人员阅读。
6. 首页和登录页适当增加品牌感与留白，提升答辩演示效果。
