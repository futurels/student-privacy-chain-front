### 1.4 术语说明

为避免后续表述歧义，本文统一采用以下术语：

- **存证**：将数据摘要、时间戳、签名或相关元数据记录到区块链中，形成可信凭证。
- **授权共享**：数据所有者或管理方对数据访问方授予可控的数据使用权限。
- **追溯审计**：对数据的提交、审批、访问、共享、回收等全过程进行记录和查询。
- **链上数据**：存储在联盟链中的摘要、授权记录、审计日志等轻量级可信信息。
- **链下数据**：原始隐私数据、加密文件、附件等，存储于数据库或 IPFS。
- **零知识证明**：在不暴露原始内容的前提下证明数据具备某种真实性或满足某种条件的隐私保护技术。

## 2. 项目背景与建设意义

在高校数字化治理持续推进的背景下，学生隐私数据已经成为教学管理、学籍管理、奖助评审、身份认证、跨部门协同和跨校验证中的关键资源。学生的身份证号、成绩、学籍信息、健康报告等数据具有高敏感、高价值、强合规属性，一旦发生泄露、越权访问、伪造篡改或责任不清，将直接影响学生隐私权益、高校治理能力及第三方核验可信度。当前传统中心化管理模式普遍存在数据孤岛、共享链路长、审计困难、责任边界不清、信任依赖单点机构等问题，难以满足“可用、可控、可信、可追溯”的新型数据治理需求。

本课题拟以区块链“不可篡改、去中心化、可追溯”的特性为基础，结合 Java、Hyperledger Fabric、智能合约、IPFS、加密与隐私保护技术，构建覆盖“数据存证、授权共享、追溯审计、隐私保护”四大核心模块的业务系统，实现高校学生隐私数据在采集、存储、共享、访问、审计、撤销等全生命周期中的可信治理。这一系统既具有工程实现价值，也具有教育数据安全治理的研究意义。

## 3. 项目目标分析

### 3.1 总体目标

系统总体目标是设计并实现一套面向高校场景的学生隐私数据可信存证与安全共享系统，通过联盟链构建可信底座，通过链上链下协同实现隐私数据可验证、可授权、可追溯、可审计、可回收。

### 3.2 业务目标

系统需满足以下业务目标：

1. 解决学生隐私数据存证不规范的问题。
2. 解决数据共享过程缺乏安全控制的问题。
3. 解决数据使用链路中责任难追踪的问题。
4. 实现学生隐私数据的全生命周期管理。
5. 支持高校内部跨部门协同与后续跨校扩展。 

### 3.3 技术目标

系统需实现以下技术目标：

1. 基于 Hyperledger Fabric 搭建联盟链底层架构。
2. 集成智能合约机制以承载存证、授权、撤销和审计逻辑。
3. 使用 IPFS 实现非结构化文件的分布式存储。
4. 形成“链上存证 + 链下加密 + IPFS 文件存储”的协同方案。
5. 实现基于角色的权限控制与安全访问管理。 

### 3.4 指标目标

系统应达到以下明确指标：

- 数据上链响应时间不高于 3 秒。
- 授权共享成功率不低于 98%。
- 支持 100+ 学生同时发起数据存证请求。
- 支持 100 用户同时在线操作。
- 数据传输采用 TLS 1.3。
- 敏感数据存储采用 AES-256。
- 满足等保三级测评基础要求。

## 4. 业务现状与问题分析

### 4.1 传统模式存在的问题

结合任务书与开题报告，高校学生隐私数据管理的典型问题主要包括：

**（1）存证不规范**
 传统系统大多依赖中心化数据库记录，缺乏不可篡改的可信存证机制，难以证明数据在某一时间点的真实性和完整性。

**（2）共享不安全**
 不同部门之间的数据共享往往依赖人工审批、表单流转、邮件传输或数据库导出，缺少可控授权、用途限制和自动回收能力。

**（3）追溯难度大**
 当发生越权访问、信息泄露或数据争议时，往往无法快速追踪操作时间、操作主体、审批链路和责任归属。

**（4）隐私保护不足**
 原始隐私数据如果直接暴露给非必要角色，容易导致超范围采集、过度共享和敏感信息泄露。

**（5）跨部门协同效率低**
 教务、学工、后勤、信息中心等部门之间缺乏统一可信底层，造成数据孤岛和重复核验。 

### 4.2 本系统拟解决的核心问题

本系统针对上述痛点，重点解决以下四类问题：

1. 如何确保学生隐私数据的真实性、完整性和不可抵赖性；
2. 如何实现数据共享的按需授权、按角色控制、按期限回收；
3. 如何实现从申请、审批、访问到回收的全过程留痕；
4. 如何在保证数据可用的同时减少原始隐私内容暴露。

## 6. 利益相关者与用户角色分析

### 6.1 利益相关者分析

系统的核心利益相关者包括：

- 学生：数据主体，也是授权发起方和个人记录查看方。
- 辅导员：负责部分学生事务核验、材料审核和业务协同。
- 教务管理员：负责学籍、成绩等结构化数据管理与审核。
- 系统管理员：负责系统用户、角色、权限、参数与平台运维。
- 高校职能部门：包括教务、学工、后勤、信息中心等链上协作节点。
- 外部验证方：后续可扩展为用人单位、兄弟院校、教育主管部门等可信验证方。 

### 6.2 用户角色定义

系统明确采用四级权限控制模型：

1. **学生**
2. **辅导员**
3. **教务管理员**
4. **系统管理员**
    其中学生仅可操作与本人相关的数据，管理员拥有权限分配、审核、配置等管理能力。

### 6.3 各角色核心需求分析

#### 6.3.1 学生角色需求

学生是系统最核心的业务主体，主要需求包括：

- 查看个人隐私数据存证记录
- 提交需要存证的数据或材料
- 查看数据上链状态、区块哈希、时间戳等摘要信息
- 发起共享授权申请
- 设置授权范围、期限、用途
- 查询授权审批进度和结果
- 撤销或终止未到期授权
- 查询个人数据访问记录与审计结果 

#### 6.3.2 辅导员角色需求

辅导员承担学生事务管理协同职能，主要需求包括：

- 查看授权范围内学生相关数据
- 审核特定类型的数据上链申请
- 审批学生数据共享申请
- 查询共享记录与审计日志
- 核验指定学生材料真实性
- 辅助处理异常申请与业务纠纷

#### 6.3.3 教务管理员角色需求

教务管理员主要面向学籍、成绩、证书等教学数据管理，需求包括：

- 录入或审核学籍、成绩等结构化数据
- 发起或确认关键数据的上链存证
- 审批面向教学场景的数据共享申请
- 根据授权范围提供数据访问或验证服务
- 查询数据来源、版本、变更与上链记录
- 处理与教务数据相关的异常与争议 

#### 6.3.4 系统管理员角色需求

系统管理员承担平台治理职责，主要需求包括：

- 用户管理、角色分配、权限配置
- 部门、组织、节点信息管理
- 平台参数维护
- 区块链节点与智能合约相关配置管理
- 审计日志查看与安全事件排查
- 字典、分类、通知、存储策略等基础配置管理
- 系统运行状态监控与异常处理

## 7. 业务流程需求分析

### 7.1 数据存证业务流程

数据存证流程是本系统的主线流程，可抽象为：

1. 数据发起方提交待存证数据；
2. 系统进行格式校验、权限校验、完整性校验；
3. 对数据计算哈希摘要并生成必要签名、时间戳信息；
4. 原始隐私数据进行链下加密存储；
5. 非结构化文件上传至 IPFS 并获取 CID；
6. 将摘要、索引、必要元数据写入区块链；
7. 返回存证结果，包括存证编号、区块哈希、交易哈希、时间戳、状态等；
8. 后续支持存证查询与一致性校验。 

### 7.2 授权共享业务流程

授权共享流程需覆盖申请、审核、执行、回收全生命周期：

1. 学生或有权主体发起共享申请；
2. 填写共享对象、共享数据范围、共享用途、有效期限；
3. 管理员或审核角色进行审批；
4. 审批通过后生成链上授权记录；
5. 被授权方在有效期内访问授权数据；
6. 系统记录每一次访问行为；
7. 到期自动回收授权，或由授权方手动撤销；
8. 所有申请、审批、访问、撤销过程均写入审计链路。

### 7.3 追溯审计业务流程

系统需支持面向事件的追溯审计流程：

1. 根据区块哈希、交易哈希、授权编号、数据编号或用户编号发起查询；
2. 获取相关数据的创建、审核、上链、访问、撤销等全流程信息；
3. 展示操作人、角色、部门、时间、动作、结果、IP 地址等关键信息；
4. 对异常访问、越权操作或共享争议形成责任链。

### 7.4 隐私保护业务流程

系统隐私保护流程应贯彻“数据可用不可见”的原则：

1. 原始敏感数据不上链；
2. 上链仅保存必要的摘要、索引和授权凭证；
3. 链下数据按需加密存储；
4. 访问时进行权限、用途、期限校验；
5. 通过零知识证明或等价隐私校验机制证明数据有效性；
6. 尽量减少明文暴露范围，确保最小必要访问。

## 8. 功能需求分析

## 8.1 功能总览

依据任务书与开题报告，系统应至少包含以下核心模块：

1. 用户与权限管理模块
2. 数据存证模块
3. 授权共享模块
4. 审计追溯模块
5. 隐私保护模块
6. 区块链管理模块
7. 文件与分布式存储模块
8. 系统基础配置模块 

------

### 8.2 用户与权限管理模块

#### 8.2.1 用户认证

系统应支持用户登录认证，能够区分不同角色身份，建立合法会话或令牌凭证，防止未授权访问。

#### 8.2.2 角色管理

系统应支持学生、辅导员、教务管理员、系统管理员四类角色的分配与维护，并允许后续扩展更细粒度角色。

#### 8.2.3 权限控制

系统应支持基于角色的访问控制，保障学生仅能访问个人数据，管理员按职责处理审核、配置与查询工作。

#### 8.2.4 用户信息管理

系统应支持维护用户基本信息、部门归属、状态、证书标识、启停用状态等。

------

### 8.3 数据存证模块

#### 8.3.1 数据类型支持

系统应支持两类数据上链：

- 结构化数据：成绩、身份证号、学籍信息等
- 非结构化数据：健康报告、证明材料、PDF 文件等

#### 8.3.2 存证申请

系统应支持用户提交待存证数据，填写数据类型、来源、描述、所属学生、提交说明等信息。

#### 8.3.3 数据摘要生成

系统应自动计算数据哈希值，用于后续一致性校验和区块链存证。

#### 8.3.4 上链登记

系统应调用区块链链码或智能合约完成存证登记，保存交易哈希、区块哈希、时间戳、上链状态等信息。

#### 8.3.5 存证记录查询

系统应支持按学生、数据类型、时间区间、状态等条件查询存证记录。

#### 8.3.6 一致性校验

系统应支持重新计算摘要并与链上记录对比，验证数据是否被篡改。

#### 8.3.7 状态管理

系统应区分草稿、待审核、已上链、失败、已作废等业务状态。

------

### 8.4 授权共享模块

#### 8.4.1 授权申请

系统应支持学生对本人隐私数据发起共享申请，填写共享对象、授权范围、用途、期限、备注等内容。

#### 8.4.2 审批处理

系统应支持辅导员、教务管理员等角色按职责审核申请，并记录审批意见。

#### 8.4.3 授权生效

审批通过后，系统应自动生成可执行授权记录，并同步上链保存授权摘要。

#### 8.4.4 授权策略

系统应支持以下授权维度：

- 按数据对象授权
- 按数据类型授权
- 按时间段授权
- 按用途授权
- 按角色或部门授权

#### 8.4.5 授权撤销

系统应支持授权方手动撤销授权，并支持智能合约自动到期回收授权。该项是课题明确提出的创新功能之一。

#### 8.4.6 授权查询

系统应支持查询授权申请状态、审批结果、当前有效授权、历史授权与撤销记录。 

------

### 8.5 审计追溯模块

#### 8.5.1 审计日志采集

系统应对登录、存证、审核、授权、访问、撤销、异常操作等关键事件进行审计记录。

#### 8.5.2 多条件检索

系统应支持通过区块哈希、交易哈希、数据编号、用户编号、授权编号等方式检索审计信息。

#### 8.5.3 责任链展示

系统应展示事件发生时间、操作主体、操作对象、操作动作、处理结果和来源信息。

#### 8.5.4 异常审计

系统应能识别越权访问、重复请求、非法操作等异常事件，为安全治理提供依据。

------

### 8.6 隐私保护模块

#### 8.6.1 数据加密

系统应对敏感数据进行链下加密存储，满足 AES-256 加密要求。

#### 8.6.2 传输保护

系统在客户端、服务端、区块链节点和存储服务之间的数据传输应采用 TLS 1.3 保护。

#### 8.6.3 零知识证明或等价隐私校验机制

任务书明确要求集成零知识证明技术，实现“数据有效性验证且不泄露内容”。在工程实现层面，可将其设计为：

- 基础版：数据哈希校验 + 最小暴露 + 访问控制
- 增强版：面向特定字段或证明场景的零知识证明机制
   该需求在毕业设计中属于亮点功能，但也属于实现难点。

#### 8.6.4 最小必要原则

系统访问控制必须遵循最小权限原则，任何角色仅可访问完成业务所必需的数据范围。

------

### 8.7 文件与分布式存储模块

#### 8.7.1 文件上传

系统应支持非结构化数据文件上传，如健康报告、证明文档、附件材料等。

#### 8.7.2 IPFS 存储

文件上传后应支持写入 IPFS 并获取内容标识 CID，作为链下文件索引信息。

#### 8.7.3 元数据管理

系统应保存文件名、类型、大小、摘要、CID、上传人、上传时间、加密状态等元数据。

#### 8.7.4 文件访问控制

系统应对文件下载、预览、验证等动作执行权限校验与审计记录。

------

### 8.8 区块链管理模块

#### 8.8.1 联盟链节点规划

系统底层联盟链至少包含教务、学工、后勤、信息中心 4 个部门节点，以支撑跨部门协同。

#### 8.8.2 链码/智能合约管理

系统应支持存证合约、授权合约、审计记录等核心逻辑的部署与调用。

#### 8.8.3 交易结果回传

系统需能够将上链结果返回给应用层，用于界面展示和业务确认。

#### 8.8.4 链上链下协同

系统需保障链上摘要与链下原始数据之间的一致性映射关系。

------

### 8.9 系统基础配置模块

系统应具备基础配置能力，包括但不限于：

- 数据类型字典管理
- 部门信息管理
- 参数配置
- 审核流程配置
- 存储策略配置
- 日志保留策略配置
- 节点与合约参数维护
   该模块虽然在任务书中未逐项展开，但从系统落地角度属于必需支撑能力。此处为基于课题目标的合理补充分析。

### 9.2 重点用例说明

#### 用例2：学生发起共享授权

- 参与者：学生、审核员、系统

#### 用例3：管理员审计追溯

- 后置条件：完成责任界定与问题排查

#### 用例4：系统自动回收到期授权

- 参与者：系统、智能合约
- 前置条件：授权已到期
- 基本流程：定时检测 → 识别到期授权 → 回收权限 → 记录日志
- 后置条件：到期授权失效，不可继续访问

### 10.2 数据分类分析

2. **隐私核心数据**：身份证号、成绩、学籍、健康报告等；

### 10.3 链上链下数据划分原则

为兼顾性能与隐私保护，系统应遵循以下原则：

- 链上只保存摘要、索引、时间戳、授权凭证、审计摘要；
- 链下保存原始明文或密文数据、文件实体及可恢复业务信息；
- 非结构化文件优先通过 IPFS 存储，并以 CID 关联；
- 数据库作为业务查询与快速索引支撑层。

## 12. 非功能需求分析

## 12.1 安全性需求

系统安全性是本课题的核心要求，应覆盖以下方面：

### 12.1.1 身份认证安全

系统应确保用户身份真实可靠，未授权用户不得访问平台敏感功能。

### 12.1.2 权限控制安全

系统应实现四级权限控制，并严格区分各角色操作边界，特别是学生仅能操作个人数据。

### 12.1.3 数据存储安全

敏感数据必须采用 AES-256 加密存储，避免明文泄露。

### 12.1.4 数据传输安全

客户端、后端、链节点和存储服务之间的传输应支持 TLS 1.3。

### 12.1.5 不可篡改性

链上存证数据一旦写入不得被随意更改，任何修改应通过新增记录体现。

### 12.1.6 可追责性

关键操作必须可回放、可定位、可问责。

------

## 12.2 性能需求

依据材料，系统性能要求包括：

- 数据上链响应时间 ≤ 3 秒；
- 授权共享成功率 ≥ 98%；
- 支持 100+ 学生并发发起存证请求；
- 支持 100 用户同时在线；
- 查询与审计功能具备可接受的交互响应速度。

------

## 12.3 可靠性需求

系统应具备以下可靠性特征：

- 关键业务操作失败后具备可回滚或可补偿能力；
- 链上交易失败、IPFS 写入失败、数据库异常时可进行错误处理；
- 保证链上链下数据映射关系不丢失；
- 审计日志不得随意删除或修改。
   这些要求属于结合区块链与业务系统融合场景作出的可靠性补充分析。

## 16. 需求分析中的关键难点与风险

### 16.1 区块链与传统 Web 系统融合难

系统需要完成 Spring Boot 后端、Vue 前端、Fabric 区块链、IPFS 存储之间的联动，存在跨技术栈集成复杂度高的问题。

### 16.2 智能合约与技术路线表述存在统一问题

材料中同时提到了 Hyperledger Fabric 与 Solidity。严格来说，Fabric 常见实现是 Chaincode（Go/Java/Node.js），而 Solidity 更常见于以太坊生态。因此在正式实现阶段，建议你在论文和设计文档中统一表述为：
 **“底层采用 Hyperledger Fabric 联盟链，智能合约逻辑采用 Fabric 链码实现；若提及 Solidity，则作为智能合约技术背景或对比说明，而非本系统唯一实现语言。”**
 这是基于材料内容识别出的一个重要需求澄清点，后续必须统一，否则会影响方案严谨性。

### 16.3 零知识证明落地难度较高

任务书要求集成零知识证明，但毕业设计周期内完整实现通用型零知识证明框架难度较大，建议在需求层面保留为增强目标，在实现层面采用“基础可落地方案 + 可扩展接口”的方式处理。

### 16.4 性能与安全的平衡难

在并发、加密、上链、审计全都开启的情况下，如何同时满足 3 秒内上链和 100+ 并发目标，是设计和测试阶段的重要挑战。

## 18、技术路线与版本建议

从当前官方版本和毕业设计的稳定性来看，后端建议采用 **JDK 21 + Spring Boot 3.5.13 + Spring Security 6 + MyBatis 3.5.16 + springdoc-openapi 2.8.16**。这样选的原因是：Spring Boot 3.5.13仍处于稳定线，且 Spring 官方文档同时列出 3.5.13 与 4.0.5 为稳定版本；而 springdoc 官方明确说明 2.8.x 对应 Spring Boot 3.5.x，当前建议使用 2.8.16。虽然 Oracle 当前最新 LTS 已经是 JDK 25，JDK 21 已变为上一代 LTS，但对毕业设计而言，JDK 21 的生态兼容性通常更稳，更适合做论文与工程落地的统一基线。

前端建议采用 **Vue 3.5.x** 稳定线，而不是追 3.6 beta。Vue 官方博客已经发布 3.5，GitHub 官方仓库当前也仍处于 3.5 稳定线提交，3.6 仍为预发布版本，因此毕业设计中写成 Vue 3.5.x 更稳妥。

区块链层建议采用 **Hyperledger Fabric 2.5.x LTS，推荐 2.5.15**。Fabric 官方文档明确说明 2.5.x 是当前 LTS 版本，安装文档也给出了 2.5.15 的下载方式；同时，Fabric 官方从 v2.4 开始建议客户端应用使用 Fabric Gateway API，Java 客户端也在推荐范围内。

这里还需要特别统一一个论文口径：你的任务书中同时出现了 **Hyperledger Fabric** 和 **Solidity**，但 Fabric 实际工程中通常使用的是 **Chaincode/链码**，而不是以太坊语境下的 Solidity 作为主实现语言。任务书确实写到了 Solidity，但从系统真实落地角度，应统一表述为“底层采用 Hyperledger Fabric，链上业务规则采用 Fabric 链码实现；Solidity 作为智能合约技术背景或对比技术出现”。这样最严谨，也最不容易在答辩时被老师追问。  

数据库建议采用 **MySQL 8.4 LTS**。因为 MySQL 官方已经明确说明 8.0 在 2026 年 4 月到达 EoL，并建议用户升级到 8.4 LTS；当前 8.4 文档已覆盖到 8.4.8。若你当前代码和脚本已经建立在 8.0 上，开发阶段可以先维持 8.0，但论文和正式设计说明里更推荐写成 8.4 LTS。

分布式文件存储建议采用 **IPFS Kubo 0.40.1**。IPFS 官方文档说明 Kubo 是最广泛使用的实现；GitHub 官方发布页显示当前最新版本为 0.40.1。若系统采用 RPC 方式与 Kubo 节点交互，IPFS 文档也明确提醒 RPC API 具有管理级权限，应仅绑定本机或内网，不得直接暴露到公网。

因此，最终推荐口径为：**JDK 21、Spring Boot 3.5.13、Spring Security 6、MyBatis-Plus 3.5.16、springdoc-openapi 2.8.16、Vue 3.5.x、Hyperledger Fabric 2.5.15、MySQL 8.4 LTS、IPFS Kubo 0.40.1、Docker 29.x**。其中链码建议优先使用 Go 或 Fabric 官方推荐方案实现，应用侧通过 Java Gateway 调用。
